# Отчёт о текущем состоянии работы

**Тема НИР:** Подготовка корпуса текстовых данных статей, разработка пайплайна предобработки и исследование методов векторизации текстов.

**Цель:** подготовка текстового датасета, разработка пайплайна предобработки и исследование методов векторизации для анализа семантического пространства текстов.

**Дата формирования отчёта:** 26.06.2026 · **Статус:** этапы 2–5 реализованы, все методы векторизации прогнаны end-to-end.

---

## 1. Введение

В рамках работы реализован исследовательский прототип конвейера обработки текстов:
**сбор корпуса → предобработка → векторизация → анализ семантического пространства**
(снижение размерности + визуализация). Конвейер оформлен как Python-пакет `textvec`
с единым интерфейсом командной строки и зафиксированными параметрами экспериментов
для воспроизводимости.

Корпус — англоязычные научные статьи из открытых источников **ArXiv** и **PubMed**.
Лемматизация реализована для английского языка (spaCy); язык вынесен в конфигурацию и
может быть переключён на русский. Настоящий отчёт документирует выполнение задач этапов
2–5 рабочего плана-графика.

## 2. Этап 2. Подготовка и анализ корпуса текстовых данных

### 2.1. Источники и метод сбора

Реализован модуль сбора (`textvec.corpus`) с двумя загрузчиками:

- **ArXiv** — через публичный API (метаданные + аннотации), запросы по тематическим категориям;
- **PubMed** — через NCBI Entrez E-utilities (`esearch` + батчированный `efetch`).

Каждый документ приводится к единой схеме и сохраняется в форматах **CSV и JSON**.
Сопутствующие метаданные: идентификатор, источник, тематика, заголовок, аннотация,
авторы, дата публикации, категории, ссылка, язык.

### 2.2. Объём и характеристики корпуса

| Показатель | Значение |
|---|---|
| Всего документов | **15 431** |
| По источникам | ArXiv — 9 618; PubMed — 5 813 |
| Тематик | 8 |
| Длина документа (слов) | мин. 12, медиана 218, среднее 221, макс. 1 182 |
| Длина (символов) | среднее 1 654, макс. 8 804 |

**Распределение по тематикам:** machine_learning 2000, astrophysics 2000,

## 3. Этап 3. Пайплайн предобработки текстов

Реализован модуль `textvec.preprocessing` со следующими стадиями (в порядке применения):

1. **Загрузка** из различных форматов (csv, json, txt, html, pdf).
2. **Очистка** от HTML-разметки, URL, e-mail, служебных символов.
3. **Нормализация**: приведение регистра, удаление пунктуации, схлопывание пробелов.
4. **Токенизация** (spaCy).
5. **Лемматизация** (английский, spaCy `en_core_web_sm`).
6. **Удаление стоп-слов** и коротких токенов; формирование очищенного корпуса.

Для полного текста дополнительно применяется структурная чистка: **срез списка
литературы** и **склейка переносов** (де-дефисация). Для длинных документов реализован
**chunking** (окна по словам + усреднение векторов) на стадии векторизации.

Корректность подтверждена модульными тестами (очистка, де-дефисация, срез references,
сквозной прогон) — **11 тестов проходят**. Обработка корпуса в 15 431 документ заняла ≈ 3 мин.

## 4. Этап 4. Методы векторизации текстов

### 4.1. Реализованные методы

| Группа | Методы | Реализация |
|---|---|---|
| Разрежённые | TF-IDF (+ TruncatedSVD / LSA) | scikit-learn |
| Плотные (классические) | FastText, Doc2Vec | gensim |
| Нейросетевые | BERT, Sentence-BERT, BGE-M3, E5, GTE | sentence-transformers |

Реализованы все методы из технического задания. Для обеспечения масштабируемости
TF-IDF не материализует плотную матрицу, а проходит через `TruncatedSVD`; длинные
документы обрабатываются с chunking; вычисления автоматически используют GPU при наличии.

### 4.2. Запущенные эксперименты (основной корпус, 15 431 док., вариант *abstract*)

| Метод | Размерность | Время, с | Примечание |
|---|---|---|---|
| TF-IDF | 300 | 8.1 | LSA, доля объяснённой дисперсии 0.26 |
| Doc2Vec | 200 | 43.2 | 40 эпох |
| FastText | 200 | 139.2 | усреднение по словам, 20 эпох |
| SBERT | 384 | 381.6 | all-MiniLM-L6-v2 |
| BERT | 768 | 1788.5 | bert-base-uncased, mean-pool, MPS |
| E5 | 768 | 1917.3 | multilingual-e5-base, MPS |
| GTE | 768 | 1453.5 | gte-base, MPS |
| BGE-M3 | 1024 | 6862.7 | BAAI/bge-m3, MPS |

Параметры каждого запуска фиксируются в `manifest.json` и `*.meta.json`.
Все 8 методов из технического задания прогнаны на трёх вариантах данных
(`abstract`, `full_text`, `abstract_subset`). Нейросетевые модели выполнялись на Apple MPS.

## 5. Этап 5. Анализ семантического пространства

### 5.1. Метод анализа

Для каждого набора эмбеддингов:
- строятся **2D-проекции** методами **PCA** и **UMAP** (косинусная метрика);
- вычисляются описательные метрики разделимости тем относительно известных
  тематических меток: **силуэт** (cosine) и **kNN-чистота тем** (доля ближайших
  соседей той же тематики). Это не кластеризация, а оценка структуры пространства.

### 5.2. Результаты (основной корпус, 15 431 док.)

| Метод | Силуэт (cosine) | kNN-чистота тем |
|---|---|---|
| **SBERT** | **0.163** | **0.849** |
| **GTE** | 0.118 | **0.844** |
| E5 | 0.104 | 0.819 |
| BGE-M3 | 0.071 | 0.802 |
| TF-IDF | 0.079 | 0.795 |
| BERT | 0.106 | 0.753 |
| FastText | 0.070 | 0.780 |
| Doc2Vec | 0.003 | 0.563 |

Лучшую тематическую разделимость демонстрирует **SBERT**; **GTE** близок по kNN-чистоте
(0.844). Проекция
`results/figures/abstract/sbert_umap.png` показывает осмысленную структуру:
изолированные кластеры astrophysics и oncology; объединение близких областей —
медицинских (cardiology + neurology) и компьютерных (machine_learning +
computation_language + cryptography_security), что соответствует семантике тематик.

### 5.3. Интерпретация

- Контекстные эмбеддинги (SBERT, GTE, E5) превосходят классические и BERT-baseline
  по тематической связности ближайших соседей.
- **GTE** и **E5** — сильные альтернативы SBERT (kNN 0.844 и 0.819); **BGE-M3** на
  коротких аннотациях уступает им (0.802), несмотря на большую размерность.
- **BERT** (mean-pooled) заметно слабее fine-tuned sentence-моделей (kNN 0.753).
- TF-IDF (LSA) даёт неожиданно высокую kNN-чистоту (0.795) — лексика научных тем
  достаточно специфична; однако силуэт низкий (пространство «размазано»).
- Doc2Vec на данном корпусе показал слабую разделимость.

## 6. Сравнительный эксперимент: аннотация vs полный текст

Чтобы оценить влияние объёма текста, собрана **полнотекстовая подвыборка — 300 документов**
(6 тем × 50; ArXiv — 250, PubMed/PMC OA — 50; в среднем ≈ 7 000 слов/документ).
Сравнение проведено на **одних и тех же документах**: вариант `full_text` (заголовок +
полный текст) против `abstract_subset` (заголовок + аннотация).

**kNN-чистота тем (одни и те же 300 документов):**

| Метод | full_text | abstract |
|---|---|---|
| **SBERT** | **0.770** | **0.755** |
| **GTE** | 0.761 | **0.755** |
| E5 | 0.714 | 0.725 |
| TF-IDF | 0.698 | 0.632 |
| BGE-M3 | 0.680 | 0.665 |
| BERT | 0.679 | 0.691 |
| Doc2Vec | 0.616 | 0.652 |
| FastText | 0.607 | 0.478 |

**Вывод.** Полный текст **умеренно улучшает** тематическую разделимость для SBERT и TF-IDF
(больше контекста и специфичной лексики), тогда как для Doc2Vec/FastText эффект смешанный.
Проекции `results/figures/full_text/sbert_umap.png` и
`results/figures/abstract_subset/sbert_umap.png` визуально схожи; у полного текста
кластеры чуть плотнее (силуэт SBERT 0.238 против 0.181).

SBERT на полном тексте обработан с chunking (300 документов → 6 162 окна, ≈ 181 с на CPU).

## 7. Ограничения и допущения

- **Полнотекст PubMed.** Свободный полный текст доступен только для подмножества
  **PMC Open Access**; свежие статьи (neurology, oncology в нашей выборке) под эмбарго,
  поэтому полнотекстовое покрытие PubMed низкое. Это ограничение источника, не пайплайна.
- **Аппаратные ресурсы.** Первые эксперименты (TF-IDF, Doc2Vec, FastText, SBERT) выполнены
  на CPU; BERT, BGE-M3, E5, GTE прогнаны на Apple MPS. BGE-M3 на полном корпусе — ≈ 1.9 ч.
- **Кластеризация** (K-Means / Agglomerative / DBSCAN из ТЗ) в задачи этапов Семестра II
  не входит — это задел под последующую работу; на текущем этапе выполнен анализ
  распределения и разделимости.
- Предупреждение gensim (`our_dot_float`) на numpy 2.0 безвредно: эмбеддинги корректны.

## Приложение A. Воспроизводимость

```bash
cd Project && python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt && pip install -e .
python -m spacy download en_core_web_sm
python -m textvec collect
python -m textvec stats
python -m textvec vectorize --variant abstract --methods bert bge-m3 e5 gte
python -m textvec analyze --variant abstract
```

Все запуски детерминированы через `project.seed = 42`. Параметры экспериментов
сохраняются в `results/embeddings/<variant>/manifest.json`.

## Приложение B. Артефакты

- `data/processed/corpus.csv|json` — корпус с метаданными; `corpus_fulltext.csv` — полнотекстовая подвыборка.
- `data/processed/clean_<variant>.csv` — очищенные корпуса.
- `results/embeddings/<variant>/<method>.npy` — векторные представления.
- `results/figures/<variant>/<method>_{pca,umap}.png` — проекции семантического пространства.
- `results/reports/` — статистика корпуса и метрики разделимости тем по вариантам.
